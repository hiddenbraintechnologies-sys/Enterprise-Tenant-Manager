1) Database tables (Rollout control per country)
✅ country_rollouts

Controls what’s enabled for each country.

Columns

country_code (PK) → "IN" | "MY" | "UK" ...

is_active (bool) → whether the country is selectable

enabled_business_types (jsonb string[]) → allowed business codes

enabled_features (jsonb) → global feature flags (payroll/hrms/etc)

enabled_modules (jsonb string[]) → allowed modules for that country

coming_soon_message (text) → optional

Example row:

{
  "country_code": "IN",
  "is_active": true,
  "enabled_business_types": ["pg_hostel"],
  "enabled_features": { "hrms": false, "payroll": false },
  "enabled_modules": ["pg", "bookings", "customers", "invoicing"],
  "coming_soon_message": ""
}

3) API endpoints (Super Admin)
Read rollout config

GET /api/super-admin/rollouts

Update rollout config

PATCH /api/super-admin/rollouts/:countryCode

Payload example:

{
  "isActive": true,
  "enabledBusinessTypes": ["pg_hostel"],
  "enabledModules": ["pg", "bookings", "customers", "invoicing"],
  "enabledFeatures": { "hrms": false, "payroll": false },
  "comingSoonMessage": ""
}

4) Enforcement points (must be backend)
✅ During signup

When user selects country + business type:

If country is_active=false → reject with COUNTRY_COMING_SOON

If business type not in enabled_business_types → reject with BUSINESS_NOT_AVAILABLE

✅ On every request

Resolve tenant’s country_code and enforce:

if module disabled → return 403 MODULE_NOT_AVAILABLE

if feature disabled → return 403 FEATURE_NOT_AVAILABLE

This ensures even if UI shows it, backend blocks it.
