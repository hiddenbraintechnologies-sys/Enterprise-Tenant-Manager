Implement full Pricing Plan Management in Super Admin Billing UI.

Current state:
Super Admin Billing → Pricing Plans list exists, but only Activate/Deactivate is supported.
We need a full Plan Builder: create plan + edit plan + configure features/limits safely.

================================================
1) Data model expectations
================================================
We already have billingPlans table (or similar) with code, name, tier, price, currency, country, isActive, and metadata jsonb.

If any of these are missing, add via migration:
- description (string, nullable)
- countryCode (string) e.g., "IN", "MY", "UK", "AE"
- currencyCode (string) e.g., "INR"
- interval (string enum: "month" | "year")
- basePrice (numeric/decimal)
- featureFlags jsonb  (e.g. { gst_features:true, whatsapp_automation:false, unlimited_records:false })
- limits jsonb (e.g. { users:3, records:500, customers:200 })
- isPublic boolean (default true) -> shown to tenants
- sortOrder int (default 100)
- version int (default 1)
- archivedAt timestamp nullable

Rules:
- code must be unique (e.g., india_basic)
- plans can be archived instead of deleted

================================================
2) Backend APIs (Super Admin only)
================================================
Add routes under /api/admin/billing/plans

RBAC:
requireAuth + requireSuperAdminOnly() OR requirePermission(MANAGE_PRICING_PLANS)

Endpoints:
1) GET /api/admin/billing/plans?country=IN
   - returns all plans (active/inactive), sorted by sortOrder

2) POST /api/admin/billing/plans
   - create plan with validation:
     - required: code, name, tier, countryCode, currencyCode, interval, basePrice
     - featureFlags defaults to {}
     - limits defaults to {}
   - audit log: PLAN_CREATED

3) PATCH /api/admin/billing/plans/:planId
   - edit allowed fields:
     name, description, basePrice, interval, isPublic, sortOrder, isActive,
     featureFlags, limits
   - Validation:
     - basePrice >= 0
     - limits values >= 0
   - audit log: PLAN_UPDATED (before/after)

4) POST /api/admin/billing/plans/:planId/archive
   - archives the plan (archivedAt = now, isActive=false, isPublic=false)
   - must block if any active tenant is currently on that plan
   - audit log: PLAN_ARCHIVED

Important:
- Never break tenants currently subscribed:
  - If plan is edited, existing tenants continue normally.
  - If you want safer behavior, implement plan version bump:
    - PATCH can optionally create a new version row and keep old row archived.
    - For MVP, direct edit is ok, but block edits to code/country/currency.
